## 2. 测试目标

本轮测试的目标不是单纯跑通某一个命令，而是覆盖下面四类能力：

- 远程 bridge 的注册、路由、回包、超时回收
- CLI 到 bridge 的远程调用链路
- 扩展到浏览器的执行链路
- 远程文件注入与试点 adapter 的真实业务链路

## 3. 测试分层

建议按下面四层推进，不要直接跳到端到端联调。

### 3.1 模块单测

目标：

- 保证每个模块自己的输入输出稳定

覆盖范围：

- Bridge
  - 注册
  - 鉴权
  - `clientId` 注册与复用
  - `GET /api/clients`
  - `POST /api/command`
  - pending 超时回收
- CLI transport
  - 本地 / 远程 transport
  - 错误文案
  - `opencli clients`
- 扩展
  - 配置保存
  - 注册
  - 心跳
  - 重连
  - `set-file-input-remote(memory)`
- Adapter
  - 小红书发布的本地 / 远程两条上传分支

### 3.2 定向集成测试

目标：

- 验证相邻模块之间能连通

覆盖范围：

- CLI -> bridge
- bridge -> extension mock
- extension -> 标准 `input[type=file]` 页面
- adapter -> `IPage.setRemoteFileInput()`

### 3.3 真实联调

目标：

- 验证真实机器、真实浏览器、真实站点

覆盖范围：

- 一台机器部署 bridge
- 一台本机 Chrome 装扩展并连接 bridge
- CLI 在本机或另一台机器执行
- 指定 `clientId` 执行真实命令

### 3.4 回归测试

目标：

- 确保新改造没有打坏旧链路

覆盖范围：

- 不带 `OPENCLI_REMOTE_URL` 时，本地 daemon 模式照常可用
- 本地 `setFileInput` 路径仍可用
- 非上传型命令在远程模式下可用

### 4.2 基础人工联调

建议按顺序做：

1. 启动 bridge
2. 扩展配置 `backendUrl` 和 `token`
3. 确认 popup 显示 `clientId`
4. 由调用方明确指定目标 `clientId`
5. 先跑基础命令
6. 再跑文件注入和试点 adapter

### 4.3 基础命令联调项

先验证不带上传的能力：

- `navigate`
- `exec`
- `cookies`
- `tabs`

目标：

- 证明 bridge、CLI、扩展、Chrome 执行链路已经贯通

### 4.4 远程文件注入联调项

建议先用标准测试页验证：

- 标准 `input[type="file"]`
- 小文件
- 中等文件
- 超过 `hardMemoryBytes` 的文件

重点检查：

- 小文件是否成功
- 中等文件是否返回 warning
- 超阈值是否明确失败
- 是否没有全局缓存字节内容

### 4.5 试点 adapter 联调项

当前试点：

- [publish.ts](/Users/samuel/Projects/SkillProjects/opencli/src/clis/xiaohongshu/publish.ts)

验证目标：

- 本地模式下本地路径上传仍可用
- 远程模式下远程图片 URL 可用
- 远程模式下本地路径会明确报错
- 错误文案能区分：
  - selector 问题
  - 注入失败
  - 模式不支持
  - 阈值失败

## 5. 重点验证项

联调时必须关注下面这些点：

- 多客户端同时在线时，命令是否严格命中指定 `clientId`
- 扩展断线重连后，原 `clientId` 是否稳定复用
- `memory` 模式是否符合阈值预期
- 小红书试点在远程模式与本地模式是否都行为正确
- 旧本地模式是否没有回归

## 6. 部署总览

当前部署对象有三类：

- `remote-bridge`
- CLI
- 浏览器扩展

小红书试点 adapter 不需要单独部署，它跟随 CLI 一起发布。

## 7. Remote Bridge 部署方案

代码位置：

- [main.ts](/Users/samuel/Projects/SkillProjects/opencli/src/remote-bridge/main.ts)
- [server.ts](/Users/samuel/Projects/SkillProjects/opencli/src/remote-bridge/server.ts)

### 7.1 开发环境启动

```bash
OPENCLI_REMOTE_BRIDGE_TOKEN=your-token npm run remote-bridge:dev
```

可选端口：

```bash
OPENCLI_REMOTE_BRIDGE_TOKEN=your-token OPENCLI_REMOTE_BRIDGE_PORT=19826 npm run remote-bridge:dev
```

### 7.2 生产环境启动

先构建：

```bash
npm run build
```

再启动：

```bash
OPENCLI_REMOTE_BRIDGE_TOKEN=your-token npm run remote-bridge:start
```

### 7.3 环境变量

- `OPENCLI_REMOTE_BRIDGE_TOKEN`
- `OPENCLI_REMOTE_BRIDGE_PORT`

### 7.4 部署建议

第一版建议：

- 单进程 Node 服务
- 放在一台固定 Linux 主机
- 前面挂 Nginx 或 Caddy 做 HTTPS / WSS 反代
- 暂不引入数据库、Redis、消息队列

### 7.5 反向代理建议

建议统一暴露：

- `https://bridge.example.com/health`
- `https://bridge.example.com/api/clients`
- `https://bridge.example.com/api/command`
- `wss://bridge.example.com/agent`

### 9.2 加载方式

第一版建议使用 Chrome 的 unpacked extension 方式加载：

- 打开 Chrome 扩展页面
- 开启开发者模式
- 加载 `extension/` 目录

### 9.3 配置项

在扩展 popup 中配置：

- `backendUrl`
- `token`

连接成功后应显示：

- 连接状态
- `clientId`

### 9.4 运行时行为

扩展启动后会：

1. 读取 `chrome.storage.local`
2. 连接远程 bridge
3. 发送 `register`
4. 获取 `clientId`
5. 周期性发送 `heartbeat`
6. 断线后自动重连

## 10. 试点 Adapter 部署与使用约束

当前试点：

- [publish.ts](/Users/samuel/Projects/SkillProjects/opencli/src/clis/xiaohongshu/publish.ts)

### 10.1 本地模式

`--images` 传本地路径：

```bash
opencli xiaohongshu publish --title "标题" "正文" --images /path/a.jpg,/path/b.jpg
```

### 10.2 远程模式

`--images` 第一版只支持远程 URL：

```bash
opencli \
  --remote-url https://bridge.example.com \
  --token your-token \
  --client cli_xxx \
  xiaohongshu publish \
  --title "标题" \
  "正文" \
  --images https://oss.example.com/a.jpg?sign=1,https://oss.example.com/b.jpg?sign=2
```

### 10.3 当前约束

- 远程模式不支持本地路径
- 本地模式不支持远程 URL
- 不支持本地路径和远程 URL 混传
